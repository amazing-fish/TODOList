# 新增任务默认截止时间严格为一小时后：设计

## 目标

新增任务启用截止时间时，默认值始终来自本地当前时间加一小时，并按界面可见的分钟精度保存。日期与时间必须来自同一个目标时刻，避免夜间跨午夜后重新拼回今天。

## 非目标

- 不修改编辑已有任务的时间回显与无损保存规则。
- 不调整滚轮、上下键、日期下拉、提醒偏移或延后语义。
- 不引入自然语言时间、快捷预设或可配置默认时长。

## 方案比较

### 方案 A：提取纯计算函数（采用）

用一个无副作用函数接收 `QDateTime`，沿绝对时间线计算 `now + 1 hour`，再通过时间线运算将秒和毫秒归零。新增对话框从返回值同时设置 `QDateEdit` 和 `QTimeEdit`，并在用户未修改可见日期和分钟时保留同一目标时刻的 UTC 值。

优点是普通时段、凌晨和跨午夜可以用固定时钟直接测试；日期与时间不会再次分离。代价是增加一个很小的模块内函数。

### 方案 B：在 `_build_ui` 中直接计算

直接把现有代码改为使用完整的 `now.addSecs(3600)`。

改动行数最少，但确定性测试需要替换 Qt 当前时间或只测试控件结果，测试边界不清晰。

### 方案 C：给对话框注入时钟

为 `TaskEditDialog` 增加 clock 参数，再由对话框计算默认值。

测试最灵活，但为了单一默认值扩大了公开构造接口，不符合本次最小修复目标。

## 设计

新增模块内函数 `_default_due_datetime(now_qdt)`：

1. 对传入的本地 `QDateTime` 加 3600 秒；
2. 保留计算结果的日期、小时和分钟；
3. 沿时间线减去秒和毫秒，避免夏令时结束的重复小时重新解析到较早 offset；
4. 返回同一时区下的目标 `QDateTime`。

`TaskEditDialog` 仅在新增任务时调用该函数。日期控件取目标日期，时间控件取目标时间，同时保存目标的 UTC ISO 值；只要用户未修改可见日期和分钟，保存时就直接使用该值，避免重复小时丢失 offset。删除凌晨 07:00 前改为 09:00、以及日期固定为今天的旧分支。

## 验证

- 固定 `13:42:37`，期望 `14:42:00`。
- 固定 `23:30:00`，期望次日 `00:30:00`。
- 固定 `05:30:00`，期望当天 `06:30:00`，不改为 09:00。
- 固定 `America/New_York` 夏令时结束前的第一个 `01:30:45.500`，期望保存第二个 `01:30:00` 对应的未来 UTC 时刻。
- mock 固定跨日目标，确认新增对话框的日期、时间与最终序列化值来自同一目标。
- 运行完整自动化测试、`python -m compileall todo_app` 与 `git diff --check`。
- 实际运行 GUI，确认新增、日期/时间回显、滚轮、保存、编辑与取消路径；提醒、延后和主题未改，仅做回归观察。
