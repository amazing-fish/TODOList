# 内容编辑保留延后状态设计

## 目标

保证任务延后后继续编辑内容或优先级时，新的截止时间、延后状态和通知状态保持不变；只有用户真正修改日期、时间或提醒偏移时才重置旧调度状态。

## 当前问题

延后生成的 `dueDate` 带有时区和微秒。`TaskEditDialog` 往返后会输出语义相同的 UTC 时间，并受 Qt 毫秒精度限制。当前代码直接比较 ISO 字符串，因此误判为调度发生变化。

## 方案

- 将合法的 `dueDate` 解析并统一到 UTC，以毫秒精度比较是否表示同一时刻。
- 语义相同时保留已有 `dueDate` 原值，避免无意义的格式和精度改写。
- 日期或分钟真实变化、提醒偏移变化时，继续清理 `snoozeUntil`、通知去重状态和 `lastNotifiedAt`。
- 保留“延后会推进任务截止时间”的现有产品语义，不调整数据结构和 UI 布局。

## 边界与风险

- 非法时间不使用兼容性猜测；无法解析时仍按实际字段变化处理。
- 毫秒以内差异不可由 Qt 编辑界面表达，因此视为同一时刻。
- 本次不修改日期控件层级、延后按钮文案或快速新增快捷键。

## 验证

- 回归测试覆盖时区表示差异、Qt 亚毫秒截断、真实分钟变化和提醒偏移变化。
- 实际 GUI 验证延后后的任务只编辑内容并保存后，新时间与延后状态不变。
