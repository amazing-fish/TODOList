# 协作约定

- 仓库内可见的说明、Issue、提交信息与 PR 使用中文，代码标识符和第三方术语可保留原文。
- 聚焦需求相关的最小修改，避免大范围无关格式化、顺手重构或引入未被需求需要的抽象。
- 修改用户可见行为、公共接口、配置或工作流时，同步更新相关文档与注释；仅内部实现调整且既有说明仍准确时，无需制造文档改动。
- 若涉及界面或资源文件变更，确保资源路径与常量定义保持一致；新增图标优先提供 SVG，并验证浅色与深色主题下均可辨识。
- 在最终说明中列出实际执行的验证命令及结果；无法执行某项验证时，说明原因和剩余风险。

## 锚点使用流程

- 开始改动前阅读需求与 `anchor.md` 中和本次任务相关的部分，确认技术路径、数据、交互、资源和版本约束。
- 任务范围发生变化，或实现触及新的模块与约束时，重新检查对应锚点；无需在同一范围内的每次提交前机械重复通读全文。
- 只有以下事实或约束发生变化时才更新 `anchor.md`：
  - 启动链路、模块职责或关键技术路径；
  - 数据结构、迁移规则或存储位置；
  - 资源契约、关键交互或提醒/托盘行为；
  - 构建发布流程、版本规则或当前版本。
- 仅文档整理、测试补充或不改变上述事实的内部重构，默认不更新锚点。
- 交付时明确说明“锚点已更新”或“锚点已复盘，无需更新”。

## 工作流检查表

### 开始前

1. 复盘需求和相关锚点，明确本次允许修改的文件与非目标。
2. 识别是否影响存储、UI、资源、提醒、托盘、主题或发布流程。
3. 先规划最小自动化与手工验证路径，再开始实现。

### 验证

1. 优先运行与改动直接相关的测试，便于快速定位问题。
2. 提交前运行完整自动化测试：

   ```powershell
   python -m unittest discover -s tests -v
   ```

3. 运行语法编译检查：

   ```powershell
   python -m compileall todo_app
   ```

4. GUI 改动按影响范围进行手工验证：
   - 对话框改动验证新增或编辑任务；
   - 提醒、推迟或托盘改动验证对应生命周期；
   - 主题或资源改动验证浅色、深色主题及资源降级行为。
5. 构建工作流、资源路径或 PyInstaller 参数发生变化时，执行相应打包验证；环境不具备时在 PR 中明确说明。
6. 执行 `git diff --check`，确认没有空白错误或意外格式化。

## 本地数据约束

- `todos.json` 是程序运行时生成的本地个人数据，不得提交到仓库。
- 测试不得覆盖开发者真实的 `todos.json`；应 mock 数据读写入口或使用临时目录。
- 示例数据应存放在明确的测试夹具或示例文件中，不得直接复用运行时数据文件。
- 修改待办字段时，同步检查 `storage.py` 的迁移逻辑、相关表单与展示组件，并按锚点规则判断是否更新数据约束。

## 提交前

1. 检查变更范围，排除运行数据、构建产物和其他无关文件。
2. 确认相关文档、注释与 `anchor.md` 是否仍然准确。
3. 汇总行为变化、验证结果、已知风险和未覆盖场景。
